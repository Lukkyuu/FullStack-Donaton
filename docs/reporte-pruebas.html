1. Introducción al Ecosistema de Pruebas

Para asegurar la confiabilidad, consistencia y correcto comportamiento de la plataforma humanitaria Donatón, se ha implementado un esquema robusto de pruebas automatizadas que abarca tanto el frontend de usuario (desarrollado en React y Vite) como los microservicios distribuidos del backend (desarrollados en Spring Boot).

La estrategia de testing incluye pruebas unitarias y de integración de controladores, pruebas unitarias para servicios de negocio y el mockeo completo de repositorios y comunicaciones externas (como publicación en colas RabbitMQ) para garantizar ejecuciones aisladas e independientes.

Ejemplo 4.1: Prueba de Integración Web - NecesidadControllerTest (Backend)

Este test valida mediante MockMvc que el endpoint de creación de necesidades sea accesible, procese datos JSON entrantes y devuelva la estructura DTO esperada. 

@WebMvcTest(NecesidadController.class)
@AutoConfigureMockMvc(addFilters = false)
public class NecesidadControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private NecesidadService necesidadService;

    @Test
    public void crearShouldReturnNeedResponse() throws Exception {
        NecesidadResponse response = NecesidadResponse.builder()
                .id(1L)
                .descripcion("Comida para damnificados")
                .estado(Necesidad.EstadoNecesidad.ACTIVA)
                .build();

        when(necesidadService.crear(any())).thenReturn(response);

        mockMvc.perform(post("/api/necesidades")
                .contentType(MediaType.APPLICATION_JSON)
                .content("{\"descripcion\":\"Comida para damnificados\",\"cantidad\":100}"))
                .andExpect(status().isCreated())
                .andExpect(jsonPath("$.id").value(1L))
                .andExpect(jsonPath("$.descripcion").value("Comida para damnificados"))
                .andExpect(jsonPath("$.estado").value("ACTIVA"));
    }
}

Ejemplo 4.2: Prueba de Servicio y Publicador RabbitMQ - DonacionServiceTest (Backend)

Este test unitario valida que la creación de una donación persista el objeto mediante el repositorio y dispare el evento correspondiente hacia la cola de RabbitMQ mediante el publisher configurado. 

@ExtendWith(MockitoExtension.class)
public class DonacionServiceTest {

    @Mock
    private DonacionRepository donacionRepository;

    @Mock
    private DonacionPublisher donacionPublisher;

    @InjectMocks
    private DonacionService donacionService;

    @Test
    public void crearShouldSaveAndPublishDonation() {
        DonacionRequest request = new DonacionRequest();
        request.setDescripcion("Ropa de invierno");
        request.setCantidad(10);

        Donacion savedDonacion = Donacion.builder()
                .id(1L)
                .descripcion("Ropa de invierno")
                .cantidad(10)
                .estado(Donacion.EstadoDonacion.PENDIENTE)
                .build();

        when(donacionRepository.save(any(Donacion.class))).thenReturn(savedDonacion);

        DonacionResponse response = donacionService.crear(request);

        verify(donacionRepository, times(1)).save(any(Donacion.class));
        verify(donacionPublisher, times(1)).publishDonacionCreada(any());
        assertEquals("Ropa de invierno", response.getDescripcion());
    }
}

Ejemplo 4.3: Prueba de Interfaz y Componentes - StatusBadge.test.jsx (Frontend)

Prueba unitaria en Vitest que evalúa el patrón de composición visual. Comprueba que el componente StatusBadge de React asigne dinámicamente las clases CSS correctas asociadas a la severidad del color según el estado inyectado. 

import { describe, it, expect } from 'vitest';
import { render, screen } from '@testing-library/react';
import { StatusBadge } from '../shared/components/index.jsx';

describe('StatusBadge', () => {
  const casos = [
    { status: 'PENDIENTE',   expectedClass: 'badge-amber' },
    { status: 'ACTIVO',      expectedClass: 'badge-green' },
    { status: 'COMPLETADO',  expectedClass: 'badge-blue'  },
    { status: 'CANCELADO',   expectedClass: 'badge-red'   },
  ];

  casos.forEach(({ status, expectedClass }) => {
    it(`renderiza estado "${status}" con clase "${expectedClass}"`, () => {
      render(<StatusBadge status={status} />);
      const badge = screen.getByText(status);
      expect(badge).toBeInTheDocument();
      expect(badge).toHaveClass('badge');
      expect(badge).toHaveClass(expectedClass);
    });
  });

  it('usa badge-gray para estado desconocido', () => {
    render(<StatusBadge status="DESCONOCIDO" />);
    const badge = screen.getByText('DESCONOCIDO');
    expect(badge).toHaveClass('badge-gray');
  });
});

5. Conclusiones y Plan de Aseguramiento de Calidad

El ecosistema de pruebas implementado en Donatón garantiza:

  1. No regresión: Los cambios sobre los microservicios se validan instantáneamente, reduciendo fallos colaterales al actualizar componentes comunes del ms-common.
  2. Seguridad Operativa: Módulos críticos como el control de acceso en frontend y autenticación de backend están cubiertos al 100%, bloqueando vulnerabilidades de autorización de forma temprana.
  3. Ciclos Cortos de Feedback: Las pruebas unitarias locales y de integración se ejecutan de manera ágil (bajo 32s para el backend y 3s para el frontend), facilitando la adopción de Integración Continua (CI/CD).
